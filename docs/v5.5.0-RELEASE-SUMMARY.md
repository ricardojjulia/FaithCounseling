**Date:** March 31, 2026
**Branch:** `main`
**Release type:** Minor
**Status:** In Development

## Overview

Introduces **Faithful Workflows**, a counselor-facing three-panel workspace that reviews each client's clinical data and surfaces prioritized, explainable, actionable care recommendations. The engine is deterministic вҖ” 27 hand-authored rules across 7 clinical categories, with no black-box AI decisions. Rationale is template-interpolated with real data values (e.g., "PHQ-9 = 18, above threshold вҖ” worsening trend 12вҶ’16вҶ’18").

This release also completes the **real-data wiring**: five canonical demo clients now carry enriched clinical profiles (PHQ-9 score histories, item-9 SI scores, GAD-7, PCL-5, treatment goal objects, faith profiles, no-show appointments) so the full urgency range is visible end-to-end after `pnpm demo:finalize`.

### Rules Engine вҖ” 27 Rules, 7 Categories

| Category | Rules |
| --- | --- |
| Safety | PHQ-9 severe (вүҘ20), SI item-9 (вүҘ2), PCL-5 crisis (вүҘ51), consecutive no-shows, risk keywords in notes |
| Clinical Cautions | PHQ-9 worsening trend, GAD-7 high (вүҘ15), no treatment plan, stale plan (>90d), no recent note (>30d), dx without goal |
| Session Focus | Overdue treatment goals, pending homework, pending assessment |
| Homework | No between-session homework in 2 sessions, journaling suggestion |
| Relationship | Systems goal missing for couples/family, relationship concern in note |
| Spiritual (all optional) | Biblical integration, faith-based grief support, transition prayer, faith acknowledge |
| Coordination | No insurance on file, open referral (>21d), faith referral, closing summary needed |
| Monitoring | Reassessment overdue (>90d), discharge planning (all goals met), stable progress |

Safety nodes with priority вүҘ 9 cannot be hidden or deferred. Spiritual nodes are always labeled "(Optional)" and never outrank safety or clinical nodes.

### Assessment Data Shape

Assessment history is stored as `form_submissions` rows (no `form_assignment` FK). The rules engine reads them via `GET /v1/forms/client-overview`, which `FaithWorkflowsPage.jsx` maps to:

```js
{ inventoryName, title, score, scoredAt, completedAt, item9Score }
```

PHQ-9 `item9Score` comes from `responses.selfHarm` / `responses.item9Score` in the submission's encrypted JSON.

## API Layer Changes

### `apps/api/src/db/queries/clientFaithProfiles.js`

Row mapper now exposes three fields used by the spiritual rules engine:

| New field | Source | Purpose |
| --- | --- |---|

## Demo Dataset Changes

### `ops/demo-dataset/manifest.mjs`

- Added `WORKFLOW_ENRICHMENT` block вҖ” 5 clinical profiles with assessment score arrays, goal objects, faith integration flags, no-show counts, and note-date overrides
- `buildClientTimeline` accepts `noteOverrideDaysAgo` вҖ” used to push Naomi's note to 36 days ago
- `buildClientBundle` applies enrichment: goal objects, faith integration flag, assessment history array, no-show appointments
- Isaac's session notes deliberately omit homework-assignment language so `ruleNoRecentHomework` fires

### `ops/demo-dataset/common.mjs`

- `client_faith_profiles` seeded with `faith_integration_level = 'actively_integrated'` for opted-in clients
- Treatment plan goals now stored as `{ description, status, targetDate }` objects (not plain strings) for all clients вҖ” required by `ruleOverdueGoals` and `ruleDischargePlanning`
- Enriched assessment history seeded as standalone `form_submissions` (no `form_assignments` row)
- `buildExpectedCounts` now adds `enrichedAssessmentHistory.length` per client to the submission total
- `every_client_has_default_form_submissions` invariant relaxed to `>=` вҖ” enriched clients have more distinct form keys than the 4 defaults
