# v5.4.2 Release Summary — Operations Studio AR Removal + Cache Stability

**Date:** March 31, 2026
**Branch:** `main`
**Release type:** Patch
**Status:** ✅ Validated

---

## Overview

Two targeted follow-on fixes to the Operations Studio repair from `v5.4.1`. First, the Accounts Receivable section was removed because the practice operates on an offerings-and-payment model — not insurance billing — making aging buckets, outstanding balances, and payer columns factually meaningless. Second, a script-tag cache buster was added to `operations.html` so that browsers immediately load the updated JavaScript rather than serving the previous cached version (which referenced the now-removed HTML elements and caused a runtime null-element crash).

---

## What Changed

### `apps/web/public/operations.js`

**`renderPracticeReport` — AR block removed**

The entire AR renderer was removed. The following element writes no longer execute:

```js
// Removed lines (representative)
const ar = s.accountsReceivable ?? {};
const totals = ar.totals ?? {};
el('rptArOutstanding').innerHTML = ...
el('rptArAging').innerHTML = ...      // aging bucket table
el('rptArClients').innerHTML = ...    // client count column
el('rptArCard').style.display = '';
```

Execution now falls through from Assessment Trends directly to Location Performance (`el('rptLocCard')`).

---

### `apps/api/src/index.js`

**`buildReportingOverview` — AR computation removed**

Three lines removed from the reporting overview builder:

| Removed | Was |
| --- | --- |
| `const tenantInvoices = filterByTenant(invoices, request);` | Filtered in-memory fake invoice array per tenant |
| `const aging = buildAgingReport(tenantInvoices, new Date().toISOString());` | Computed fake aging buckets from 2 hardcoded invoices |
| `accountsReceivable: aging,` | Attached fake AR data to API response |

The `buildAgingReport` function itself is **not removed** — it is still called by `GET /api/v1/billing/aging-report` at line ~6388. Only _this usage_ in `buildReportingOverview` was removed.

### `apps/web/public/operations.html`

**`rptArCard` div removed**

<!-- Removed -->

  <div id="rptArOutstanding">...</div>
  <div id="rptArAging">...</div>

### Script tag cache busted

```html
<!-- Before -->
<script defer src="/operations.js"></script>

<!-- After -->
<script defer src="/operations.js?v=5.4.2"></script>
```

## Root Cause — Why AR Was Removed

### Practice model mismatch

| Domain concept | Practice model | AR section assumed |
| --- | --- | --- |
| Revenue collection | Client pays for counseling packages (offerings) | Payer/insurance billing |
| Payment tracking | `payments` table linked to `offerings` | AR aging from invoices |
| Claims | None | CPT codes, payer codes |
| Outstanding balances | Reconciled per offering payment | Open invoice buckets |

The AR section powered its display from `buildAgingReport`, which drew from `invoices` — a hardcoded in-memory array seeded with two fictional records:

```js
{ id: 'inv-001', payer: 'BlueCross Placeholder', amount: 1500, dueDate: '2026-02-01', ... }
{ id: 'inv-002', payer: 'Self-Pay', amount: 350, dueDate: '2026-03-01', ... }
```

These values were never database-backed and never represented real practice state. Displaying them as operational data was actively misleading.

### Browser cache null-element crash

After the AR HTML elements were removed from `operations.html`, browsers that had cached the previous `operations.js` began throwing:

```text
Cannot set properties of null (setting 'innerHTML')
```

Root cause: the cached JS still attempted `el('rptArAging').innerHTML = ...`, but `rptArAging` no longer existed in the DOM. The `el()` helper returns `document.getElementById(id)`, which returns `null` for missing elements; assigning `.innerHTML` on `null` throws immediately.

**Fix:** Adding `?v=5.4.2` to the script src forces browsers to treat the URL as a new resource and bypass the cache, ensuring the updated (AR-free) JS is loaded from the server.
